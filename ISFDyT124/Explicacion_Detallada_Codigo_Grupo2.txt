Cabecera y Dependencias (Líneas 1-21)
------------------------------------------------------------------------
* Línea 12: [Authorize(Roles = "Profesor")]
  - Explicación: Es un filtro de autorización. Restringe el acceso a todo el
    controlador únicamente a usuarios que tengan el rol de "Profesor". Si un
    alumno o administrador intenta ingresar, el middleware de ASP.NET Core
    intercepta la petición y devuelve un código 403 Forbidden o redirige al Login.
  - Defensa: "Garantiza la seguridad a nivel de controlador mediante el sistema
    de control de accesos basado en roles (RBAC) integrado en ASP.NET Core".

* Líneas 15-21: Constructor e Inyección de Dependencias
  - private readonly InstitutoDbContext _context;
  - public ProfesorController(InstitutoDbContext context) { _context = context; }
  - Explicación: ASP.NET Core inyecta el contexto de la base de datos
    'InstitutoDbContext' en el constructor del controlador de forma automática
    (registrado en Program.cs). Lo almacenamos en el campo privado de solo lectura
    '_context' para realizar consultas LINQ.
  - Defensa: "Evita el uso de acoplamiento fuerte (como usar 'new InstitutoDbContext()')
    y nos permite delegar el ciclo de vida del contexto al contenedor de dependencias
    del framework, lo que facilita pruebas unitarias y la mantenibilidad".

Acción Index() - Dashboard de Cátedras (Líneas 23-65)
------------------------------------------------------------------------
* Línea 30: var docenteId = int.Parse(User.FindFirstValue(ClaimTypes.NameIdentifier));
  - Explicación: Extrae el identificador único del usuario (UsId) desde los Claims
    almacenados en la cookie de autenticación cifrada del navegador.
  - Defensa: "Obtenemos el ID del docente directamente del contexto de seguridad de la
    sesión (User.Identity), previniendo que un usuario suplante a otro modificando
    parámetros en la URL".

* Líneas 33-38: Consulta de Cátedras con LINQ (Eager Loading)
  - var misCatedras = await _context.Usuarios
        .Where(u => u.UsId == docenteId)
        .SelectMany(u => u.CarreraMaterias)
        .Include(cm => cm.Carrera)
        .Include(cm => cm.Materia)
        .ToListAsync();
  - Explicación: Filtra el usuario docente actual, navega por su relación intermedia
    'CarreraMaterias' y utiliza '.Include()' para cargar con carga ansiosa (Eager Loading)
    los objetos de Carrera y Materia relacionados en una única sentencia SQL.
  - Defensa: "Usamos Eager Loading para evitar el problema de consultas 'N+1', trayendo
    todas las entidades relacionadas (Carreras y Materias) en una sola consulta de base
    de datos optimizada".

* Líneas 41-55: Mapeo manual al DTO (HomeIndexDto)
  - Explicación: Se crea un objeto 'HomeIndexDto' y se pueblan sus listas 'Carreras'
    y 'Materias' extrayendo los datos de 'misCatedras'. Se utiliza '.Distinct()'
    para asegurar que si el docente dicta la misma materia en distintos años, la materia
    o carrera figure una sola vez en el selector visual.
  - Defensa: "El DTO (Data Transfer Object) se utiliza para desacoplar el modelo de base
    de datos físico de la interfaz web, enviando solo los datos necesarios y protegiendo
    los campos sensibles".

Acción Asistencia() - Planilla Diaria (Líneas 67-116)
------------------------------------------------------------------------
* Línea 71: public async Task<IActionResult> Asistencia(int caId, int coId, int maId, DateTime? fecha)
  - Explicación: Acción GET que recibe el ID de Carrera (caId), de Cohorte (coId),
    de Materia (maId) y la fecha de toma de asistencia.

* Líneas 73-80: Validación de Fecha Futura
  - var fechaFiltro = fecha ?? DateTime.Today;
  - if (fechaFiltro > DateTime.Today) { ... return View("Error"); }
  - Explicación: Si no se pasa una fecha, se toma el día de hoy. Luego se valida que
    no sea posterior a la fecha actual para evitar registros inválidos en el futuro.
  - Defensa: "Aplicamos una regla de negocio y validación del lado del servidor para
    evitar inconsistencias lógicas en el presentismo".

* Línea 82-84: Obtención de Módulos
  - var materia = await _context.Materias.FindAsync(maId);
  - ViewBag.CantModulos = materia?.MaCantModulos ?? 1;
  - Explicación: Busca la materia seleccionada para conocer la cantidad de módulos
    asignados (de 1 a 4) y la expone mediante 'ViewBag' para renderizar los checkboxes.

* Líneas 87-97: Cargar Alumnos de la Carrera y Cohorte
  - Carga el registro intermedio de 'CarreraCohortes' usando 'CaId' y 'CoId'.
  - Recupera los usuarios que tienen rol de Alumno (RoId == 3) asignados a ese
    'CaCoId', ordenándolos alfabéticamente por Apellido y Nombre.
  - Defensa: "Buscamos los estudiantes matriculados en la combinación carrera-año
    y ordenamos los resultados de forma alfabética para facilitar el trabajo del docente".

* Líneas 100-102: Carga de Asistencias Existentes (Precarga / Edición)
  - var asistenciasExistentes = await _context.Asistencias
        .Where(a => a.MaId == maId && a.AsFecha.Value.Date == fechaFiltro.Date)
        .ToDictionaryAsync(a => a.UsId.Value, a => a);
  - Explicación: Consulta si existen registros de asistencias tomadas hoy para esa materia.
    Lo convierte a un diccionario indexado por el ID de usuario ('UsId').
  - Defensa: "Convertir la colección de asistencias en un Diccionario en C# nos permite
    hacer búsquedas en tiempo O(1) constante en la vista al renderizar cada fila, precargando
    los estados anteriores de asistencia sin ralentizar la página".

Acción GuardarAsistencia() - Upsert Transaccional (Líneas 117-176)
------------------------------------------------------------------------
* Línea 120: public async Task<IActionResult> GuardarAsistencia(int maId, DateTime fecha, Dictionary<int, AsistenciaCrearDto> asistencias)
  - Explicación: Recibe el ID de la materia, la fecha y un Diccionario donde la clave es
    el ID del alumno (UsId) y el valor contiene si asistió y si está justificado.
  - Defensa: "El enlazador de modelos de ASP.NET Core (Model Binder) mapea los inputs del
    formulario automáticamente a un Diccionario estructurado en C#, simplificando la lógica".

* Línea 128: int proximoAsId = _context.Asistencias.Any() ? await _context.Asistencias.MaxAsync(a => a.AsId) + 1 : 1;
  - Explicación: Busca el valor de ID de asistencia más alto en la base de datos y le suma
    1. Esto se requiere porque las tablas físicas no tienen la propiedad 'IDENTITY' en SQL Server.
  - Defensa: "Como el esquema de base de datos no autogenera los IDs, el software calcula
    secuencialmente la clave primaria antes del almacenamiento garantizando la integridad de datos".

* Líneas 130-167: Bucle de Inserción o Modificación (Upsert)
  - Itera la lista de alumnos. Si el alumno está marcado como presente, la justificación
    se fuerza a false (`justificado = false`) por regla lógica.
  - Busca si existe un registro de asistencia previo (`AsistenciaExistente`).
  - Si existe: Actualiza los campos `AsPresente` y `AsJustificacion` y llama a `_context.Update()`.
  - Si no existe: Crea una nueva entidad `Asistencia` asignándole el `proximoAsId` (e incrementándolo)
    y la inserta con `_context.Asistencias.Add()`.

* Línea 170: await _context.SaveChangesAsync();
  - Explicación: Guarda los cambios acumulados en memoria.
  - Defensa: "EF Core ejecuta todas las inserciones y actualizaciones dentro de una única
    transacción SQL implícita. Si algún registro falla, la transacción entera se revierte (ACID),
    evitando que queden planillas parciales o incompletas".

Acciones Create() - Registro de Alumnos (Líneas 302-384)
------------------------------------------------------------------------
* GET Create(): Carga en `ViewBag.CarreraCohortesList` las carreras y cohortes disponibles para el dropdown.

* POST Create(UsuarioCrearDto model):
  - Verifica `ModelState.IsValid`. Si no es válido, retorna la misma vista enviando los mensajes de error.
  - Valida duplicación de DNI: `_context.Usuarios.AnyAsync(u => u.UsDni == model.UsDni)`.
    Si el DNI ya existe, añade un error con `ModelState.AddModelError("UsDni", "...")`.
  - Generación de claves manuales secuenciales:
    Busca el ID máximo para `Usuario` y para `UsuarioRol` y suma `+1` a cada uno.
  - Asigna Contraseña por Defecto: `UsContrasena = model.UsDni.ToString()`.
    Defensa: "Por políticas del proyecto, el alumno ingresa inicialmente usando su número
    de DNI como contraseña. Posteriormente podrá modificarla desde su perfil de usuario".
  - Inserta el registro del estudiante en `Usuarios` y su rol 'Alumno' en `UsuarioRoles`.

------------------------------------------------------------------------
Acciones Edit() - Modificación de Alumnos (Líneas 386-480)
------------------------------------------------------------------------
* Recupera el alumno usando el ID y valida que tenga el rol de Alumno (`RoId == 3`).
* Al procesar el POST:
  - Valida el DNI duplicado pero excluyendo al alumno que se está editando actual:
    `u => u.UsDni == model.UsDni && u.UsId != id`.
    Defensa: "Esto permite verificar que el DNI modificado no pertenezca a OTRO usuario ya
    registrado, pero no arroja error si el DNI es el del propio alumno siendo editado".

------------------------------------------------------------------------
Acción Delete() - Eliminación de Alumnos (Líneas 482-502)
------------------------------------------------------------------------
* Recibe el POST de eliminación.
* Recupera el estudiante, elimina sus roles en la tabla intermedia `UsuarioRoles` para evitar
  inconsistencias o errores de clave foránea, elimina el registro principal en `Usuarios`
  y guarda los cambios en la base de datos de manera transaccional.

PARTE 5: DIRECTRICES DE ARQUITECTURA Y PREGUNTAS CLAVE PARA LA DEFENSA
========================================================================

1. P: ¿Por qué hay un archivo de vistas en Views/Alumnos si el controlador está dentro de AdminController.cs?
   R: "ASP.NET Core resuelve las vistas buscando en la carpeta `Views/[NombreControlador]/`. 
      Como la clase controladora se llama `AlumnosController`, las vistas deben estar en 
      `Views/Alumnos/`. Al colocar la clase `AlumnosController` al final del archivo 
      `AdminController.cs`, mantuvimos el orden lógico de los controladores y al mismo tiempo 
      conservamos las vistas y las rutas `/Alumnos` intactas sin alterar el enrutamiento existente."

2. P: ¿Cómo funciona el paso de datos mediante ViewBag, ViewData y TempData?
   R: - 'ViewData' y 'ViewBag' transfieren datos efímeramente desde el controlador hacia la vista
        en la misma petición HTTP. (ViewBag es dinámico; ViewData usa diccionarios tipados).
      - 'TempData' almacena los datos de forma temporal en la sesión del servidor y sobrevive a 
        una redirección (RedirectToAction). Se autodestruye una vez que es leído en la vista
        (ej. para mostrar el mensaje exitoso 'Alumno registrado con éxito').

3. P: ¿Por qué no usamos Identity para los roles y claves de usuario?
   R: "Se implementó un esquema de autenticación por cookies personalizado acoplado a la 
      estructura de base de datos existente del instituto. Esto nos dio control total sobre las 
      tablas de usuarios y asignación de roles sin el peso adicional de las tablas por defecto
      de ASP.NET Core Identity".

4. P: ¿Cómo garantizamos la seguridad en los formularios ante ataques externos?
   R: "Mediante el uso de `@Html.AntiForgeryToken()` en las vistas y el decorador 
      `[ValidateAntiForgeryToken]` en los métodos POST de los controladores. Esto inyecta un token 
      oculto de un solo uso en la vista y en la cookie; al hacer POST, el servidor los compara y 
      verifica que provengan de nuestra web, mitigando ataques de falsificación de petición en 
      sitios cruzados (CSRF)".
